# 🤖 Service Reminder Bot

Telegram-бот для отслеживания сроков окончания оплаченных услуг (VDS-серверы, почта, домены, подписки и т.д.) и уведомления об их истечении.

---

## Обзор

Бот решает одну задачу: **не дать пропустить окончание оплаченного сервиса**. Пользователь добавляет услугу с указанием названия и даты окончания, а бот за 7, 3 и 1 день до истечения срока присылает напоминание. Управление ведётся через inline-кнопки — никаких команд запоминать не нужно. Если услуга продлена, достаточно нажать одну кнопку и выбрать готовый период (1 мес / 3 мес / 6 мес / 1 год) — или ввести дату вручную. Всё предельно просто: минимум зависимостей, файловое хранение данных, никаких баз данных.

---

## Функционал

### Управление услугами

Все действия доступны через **inline-кнопки** под сообщениями бота:

| Действие | Кнопка | Описание |
|---|---|---|
| Главное меню | `/start` | Отображает кнопки: ➕ Добавить, 📋 Мои услуги |
| Добавить услугу | `➕ Добавить услугу` | Бот запрашивает название и дату |
| Список услуг | `📋 Мои услуги` | Показывает список с кнопками к каждой услуге |
| Продлить услугу | `🔄 Продлить` | Выбор шаблона периода или ввод произвольной даты |
| Удалить услугу | `🗑 Удалить` | Удаление с подтверждением через inline-кнопку |
| Очистить всё | `🗑 Удалить все` | Очистка всего списка с подтверждением |

### Уведомления

Бот автоматически проверяет даты окончания и присылает напоминания:

| За сколько дней | Значок | Суть |
|---|---|---|
| 7 дней | 🟡 | Первое предупреждение — есть время продлить |
| 3 дня | 🟠 | Второе предупреждение — скоро истечёт |
| 1 день | 🔴 | Третье предупреждение — последний шанс |
| 0 дней | ⛔ | Услуга истекла — пора действовать |

После истечения срока (0 дней) бот отправляет **финальное уведомление** с кнопкой «Продлить». Дальше уведомления по этой услуге прекращаются до момента продления.


**Кнопки под сообщением:**

```
[ ➕ Добавить услугу ]  [ 📋 Мои услуги ]
```

### Добавление услуги (шаг за шагом через кнопки)

**Шаг 1** — пользователь нажимает `➕ Добавить услугу`:

```
📝 Введите название услуги:

Например: Timeweb VPS, Яндекс 360, Reg.ru домен
```

**Шаг 2** — пользователь вводит название (например, `Timeweb VPS`):

```
📅 Введите дату окончания для «Timeweb VPS»

Формат: YYYY-MM-DD
Например: 2026-12-31
```

**Шаг 3** — пользователь вводит дату (например, `2026-12-31`):

```
✅ Услуга «Timeweb VPS» добавлена!

📅 Окончание: 2026-12-31 (через 277 дней)
🆔 ID: 1

[ 📋 Мои услуги ]
```

**Обработка ошибок на каждом шаге:**

Неверный формат даты:
```
❌ Неверный формат даты. Используйте: YYYY-MM-DD
Попробуйте ещё раз:
```

Пустой ввод:
```
❌ Название не может быть пустым. Попробуйте ещё раз:
```

### Список услуг (`📋 Мои услуги`)

> **Примечание по реализации:** Если у пользователя больше 5 услуг, список разбивается на страницы с кнопками `◀️ Назад` / `▶️ Далее`. В рамках MVP пагинация является желательной, но не обязательной — можно выводить весь список одним сообщением, если услуг немного.

### Продление услуги (`🔄 Продлить`)

Пользователь нажимает `🔄 Продлить` рядом с нужной услугой. Вместо ручного ввода даты бот предлагает **готовые шаблоны периодов**:

```
🔄 Продление: «Timeweb VPS»

Текущая дата окончания: 2026-12-31

Выберите период продления:
```

**Кнопки шаблонов (две строки):**

```
[ 📅 1 месяц  ]  [ 📅 3 месяца ]
[ 📅 6 месяцев ]  [ 📅 1 год    ]

[ ✏️ Другая дата ]  [ ❌ Отмена   ]
```

**Расчёт новой даты:** Новая дата = текущая дата окончания + выбранный период. Например, при текущей дате `2026-12-31` и выборе «6 месяцев» новая дата будет `2027-06-30`. Используется `dateutil.relativedelta` для корректного прибавления месяцев (учитывает разную длину месяцев).

**Примеры расчётов для текущей даты `2026-12-31`:**

| Шаблон | Новая дата | Результат |
|---|---|---|
| 1 месяц | 2027-01-31 | +30 дней |
| 3 месяца | 2027-03-31 | +90 дней |
| 6 месяцев | 2027-06-30 | +181 день |
| 1 год | 2027-12-31 | +365 дней |

**Результат при выборе шаблона (например, «6 месяцев»):**

```
✅ Услуга «Timeweb VPS» продлена на 6 месяцев!

📅 Было: 2026-12-31
📅 Стало: 2027-06-30 (через 181 день)
🆔 ID: 1

[ 📋 Мои услуги ]
```

#### Произвольная дата (`✏️ Другая дата`)

Если ни один шаблон не подходит, пользователь нажимает `✏️ Другая дата`:

```
✏️ Произвольная дата для «Timeweb VPS»

Текущая дата окончания: 2026-12-31

Введите новую дату окончания:
Формат: YYYY-MM-DD
```

Пользователь вводит дату (например, `2028-03-15`):

```
✅ Услуга «Timeweb VPS» продлена!

📅 Было: 2026-12-31
📅 Стало: 2028-03-15 (через 805 дней)
🆔 ID: 1

[ 📋 Мои услуги ]
```

**Особенности продления:**
- Флаги уведомлений (`notified_7`, `notified_3`, `notified_1`, `notified_0`) **сбрасываются** — бот снова будет уведомлять за 7/3/1 день до новой даты
- При использовании шаблона дата рассчитывается от **текущей даты окончания** услуги (не от сегодняшнего дня) — это позволяет продлить даже уже истёкшую услугу на нужный период

**Обработка ошибок (произвольная дата):**

Новая дата раньше или равна текущей дате окончания:
```
❌ Новая дата должна быть позже текущей даты окончания (2026-12-31).
Введите корректную дату:
```

Неверный формат:
```
❌ Неверный формат даты. Используйте: YYYY-MM-DD
Попробуйте ещё раз:
```

### Удаление услуги (`🗑 Удалить`)

## Хранение данных (JSON-файл)

Вместо базы данных используется один файл `data.json` в корне проекта. Формат:

```json
{
  "123456789": {
    "services": [
      {
        "id": 1,
        "name": "Timeweb VPS",
        "end_date": "2026-12-31",
        "notified_7": false,
        "notified_3": false,
        "notified_1": false,
        "notified_0": false,
        "created_at": "2026-03-29"
      },
      {
        "id": 2,
        "name": "Яндекс 360",
        "end_date": "2026-04-01",
        "notified_7": true,
        "notified_3": false,
        "notified_1": false,
        "notified_0": false,
        "created_at": "2026-03-20"
      }
    ]
  },
  "987654321": {
    "services": [
      {
        "id": 1,
        "name": "Reg.ru домен",
        "end_date": "2026-06-15",
        "notified_7": false,
        "notified_3": false,
        "notified_1": false,
        "notified_0": false,
        "created_at": "2026-03-29"
      }
    ]
  }
}
```

### Структура данных

- **Ключ верхнего уровня** — `user_id` (Telegram ID пользователя, строка)
- **services** — массив услуг данного пользователя
  - **id** — уникальный идентификатор услуги в рамках пользователя (инкрементальный)
  - **name** — название услуги (строка)
  - **end_date** — дата окончания, формат `YYYY-MM-DD`
  - **notified_7/3/1/0** — флаги, отправлено ли соответствующее уведомление
  - **created_at** — дата добавления услуги

### Почему JSON, а не SQLite?

| Критерий | JSON-файл | SQLite |
|---|---|---|
| Зависимости | Нет (stdlib `json`) | Нет (stdlib `sqlite3`) |
| Сложность кода | Минимальная | Немного больше (SQL-запросы) |
| Читаемость | Открывается в любом редакторе | Нужен клиент БД |
| Бэкап | Скопировать один файл | Скопировать один файл |
| Переносимость | Максимальная | Хорошая |
| Для 1-100 услуг | Более чем достаточно | Избыточно |
| Конкурентный доступ | Не предназначен | Поддерживается |

Для задачи с небольшим объёмом данных и одним процессом бота JSON — оптимальный выбор. Если в будущем понадобится масштабирование — миграция на SQLite или PostgreSQL займёт минимальное время благодаря абстракции в `storage.py`.

### Потокобезопасность

Поскольку бот работает в одном процессе (asyncio), конкурентный доступ к файлу не возникает. Однако для подстраховки используется механизм блокировки через `asyncio.Lock`:

```python
# В storage.py
_lock = asyncio.Lock()

async def save_data():
    async with _lock:
        with open(DATA_FILE, 'w', encoding='utf-8') as f:
            json.dump(_data, f, ensure_ascii=False, indent=2)
```

Это гарантирует, что одновременное сохранение от обработчика и от планировщика уведомлений не приведёт к повреждению файла.

## Управление состоянием (ConversationHandler)

Для многошаговых действий (добавление услуги, продление) используется `ConversationHandler` из `python-telegram-bot`. Состояния:

```
CHOOSING_ACTION    → Главное меню (кнопки: Добавить, Мои услуги)
WAITING_NAME       → Ожидание ввода названия услуги
WAITING_DATE       → Ожидание ввода даты окончания
CHOOSING_PERIOD    → Выбор шаблона продления (1 мес / 3 мес / 6 мес / 1 год / другая дата)
WAITING_CUSTOM_DATE → Ожидание ввода произвольной даты (продление)
CONFIRM_DELETE     → Подтверждение удаления конкретной услуги
CONFIRM_CLEAR      → Подтверждение удаления всех услуг
```

### Схема переходов

```
/start, /help
    └─► CHOOSING_ACTION (кнопки: ➕ Добавить, 📋 Мои услуги)

CHOOSING_ACTION ─[➕ Добавить]──► WAITING_NAME
CHOOSING_ACTION ─[📋 Мои услуги]─► вывод списка услуг (остаётся в CHOOSING_ACTION)

WAITING_NAME ─[текст]──► WAITING_DATE (сохраняем название)
WAITING_DATE ─[текст]──► CHOOSING_ACTION (сохраняем услугу, показываем результат)

Из списка услуг:
    ─[🔄 Продлить]──► CHOOSING_PERIOD
    ─[🗑 Удалить]──► CONFIRM_DELETE
    ─[🗑 Удалить все]──► CONFIRM_CLEAR

CHOOSING_PERIOD ─[📅 1 мес / 3 мес / 6 мес / 1 год]──► CHOOSING_ACTION (рассчитываем дату, обновляем, сбрасываем флаги)
CHOOSING_PERIOD ─[✏️ Другая дата]──► WAITING_CUSTOM_DATE
CHOOSING_PERIOD ─[❌ Отмена]──► CHOOSING_ACTION (возвращаемся к списку)

WAITING_CUSTOM_DATE ─[текст]──► CHOOSING_ACTION (обновляем дату, сбрасываем флаги)
CONFIRM_DELETE ─[✅ Да]──► CHOOSING_ACTION (удаляем)
CONFIRM_DELETE ─[❌ Отмена]──► CHOOSING_ACTION (возвращаемся)
CONFIRM_CLEAR ─[✅ Да]──► CHOOSING_ACTION (удаляем все)
CONFIRM_CLEAR ─[❌ Отмена]──► CHOOSING_ACTION (возвращаемся)

Команда /start в любом состоянии ─► CHOOSING_ACTION (сброс диалога)
```

### Callback data для inline-кнопок

Для передачи контекста через кнопки используется формат:

```
act:<action>:<service_id>

Примеры:
act:list             — открыть список услуг
act:add              — начать добавление
act:renew:3          — показать меню продления услуги с ID 3
act:renew_period:3:1m   — продлить услугу 3 на 1 месяц
act:renew_period:3:3m   — продлить услугу 3 на 3 месяца
act:renew_period:3:6m   — продлить услугу 3 на 6 месяцев
act:renew_period:3:1y   — продлить услугу 3 на 1 год
act:renew_custom:3    — ввести произвольную дату для услуги 3
act:renew_cancel:3    — отмена продления услуги 3
act:del:3             — запросить удаление услуги с ID 3
act:del_confirm:3     — подтвердить удаление услуги с ID 3
act:del_cancel:3      — отменить удаление услуги с ID 3
act:clear             — запросить удаление всех
act:clear_confirm     — подтвердить удаление всех
act:clear_cancel      — отменить удаление всех
```

## Логика уведомлений (`notifier.py`)

Планировщик (APScheduler) запускает задачу **один раз в сутки** в заданное время (по умолчанию `09:00` по часовому поясу из `.env`).

### Алгоритм

```
1. Загрузить data.json
2. Получить текущую дату (локальную, согласно часовому поясу из конфига)
3. Для каждого пользователя в data:
   a. Для каждой услуги пользователя:
      - Вычислить days_left = (end_date - today).days
      - Если days_left == 7  и notified_7 == False → отправить 🟡, установить notified_7 = True
      - Если days_left == 3  и notified_3 == False → отправить 🟠, установить notified_3 = True
      - Если days_left == 1  и notified_1 == False → отправить 🔴, установить notified_1 = True
      - Если days_left <= 0  и notified_0 == False → отправить ⛔, установить notified_0 = True
   b. Сохранить data.json (если были изменения)
4. Услуги с notified_0 == True более не обрабатываются
5. Если пользователь продлевает услугу — все флаги notified_7/3/1/0 сбрасываются в False
```

> `python-dateutil` — нужен для корректного прибавления месяцев к дате (`relativedelta`), так как стандартный `datetime.timedelta` не умеет работать с месяцами из-за их переменной длины.

## Многопользовательский режим

Бот поддерживает **любое количество пользователей**. Данные каждого пользователя изолированы по `user_id` (ключ верхнего уровня в JSON-файле) — каждый видит и управляет только своими услугами. Никакой регистрации или авторизации не требуется: достаточно отправить боту `/start`, и бот автоматически создаст запись для нового пользователя.

---

## Безопасность

- **Токен бота** хранится в `.env` и никогда не попадает в репозиторий (`.env` добавлен в `.gitignore`)
- **Данные пользователей** хранятся в локальном файле `data.json`, не передаются третьим сторонам
- **Нет web-хуков без SSL** — бот работает в polling-режиме, не требует открытых портов
- **JSON-файл не содержит секретных данных** — только названия услуг и даты

## Деплой и развёртывание

### Вариант 1: Локальный запуск (для разработки)

```bash
# 1. Клонировать репозиторий
git clone https://github.com/<username>/service-reminder-bot.git
cd service-reminder-bot

# 2. Создать виртуальное окружение
python -m venv venv
source venv/bin/activate  # Linux/macOS
# venv\Scripts\activate   # Windows

# 3. Установить зависимости
pip install -r requirements.txt

# 4. Создать файл конфигурации
cp .env.example .env
# Отредактировать .env: вставить BOT_TOKEN, указать TZ

# 5. Запустить бота
python bot.py
```

### Вариант 2: Docker (рекомендуется для VPS)

```bash
# 1. Клонировать репозиторий
git clone https://github.com/<username>/service-reminder-bot.git
cd service-reminder-bot

# 2. Создать файл конфигурации
cp .env.example .env
# Отредактировать .env: вставить BOT_TOKEN, указать TZ

# 3. Запустить через Docker Compose
docker compose up -d

# 4. Посмотреть логи
docker compose logs -f
```

### Вариант 3: Системный сервис (systemd) на VPS

```bash
# 1. Создать файл сервиса
sudo nano /etc/systemd/system/reminder-bot.service
```

Содержимое:

```ini
[Unit]
Description=Service Reminder Telegram Bot
After=network.target

[Service]
Type=simple
User=botuser
WorkingDirectory=/opt/service-reminder-bot
EnvironmentFile=/opt/service-reminder-bot/.env
ExecStart=/opt/service-reminder-bot/venv/bin/python bot.py
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```

```bash
# 2. Активировать и запустить
sudo systemctl daemon-reload
sudo systemctl enable reminder-bot
sudo systemctl start reminder-bot

# 3. Проверить статус
sudo systemctl status reminder-bot
```
